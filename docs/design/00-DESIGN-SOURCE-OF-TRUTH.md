## Decisions D1-D5 — CLOSED (the code decided; verified live 2026-07-12)

The 2026-07-12 assessment flagged five contradictions as "open decisions". The recovered
implementation had already settled all five:

- **D1 — BFF vs plain JWT: BOTH (dual-mode), already built.** The API is a pure stateless
  JWT bearer service (ClockSkew=0, ProblemDetails 401/403); `AzureBank.Bff` is a YARP 2.3
  proxy holding the JWT server-side (`InMemoryTokenStore`) behind an HttpOnly
  `.AzureBank.Session` cookie (Secure, SameSite=Strict), with step-up PIN auth
  (AuthLevel 1/2, ADR-0008), rate limiting and security headers. CORS on both allows the
  SPA to use either mode. Verified live in both modes. *Erratum for `08`/`backend-guide`:
  neither "rejected" model lost — the API works with and without the BFF (ADR-0001/0002).*
- **D2 — Identity vs custom User: BOTH, reconciled.** `ApplicationUser : IdentityUser<Guid>`
  (+AzureTag, PinHash, …) on standard `AspNetUsers` tables plus domain tables
  (`Accounts`, `Transactions`, `RefreshTokens`). Passwords use the Identity hasher (PBKDF2);
  **Argon2id (Konscious) is used for PINs** with OWASP profiles, PHC format, FixedTimeEquals.
  *Erratum for ADR-0003: it prescribed Argon2id for passwords; the code applies it to PINs.*
- **D3 — Wire contract: DECIDED.** Success envelope `ApiResponse<T>` `{data, message}`;
  errors are RFC 9457 ProblemDetails everywhere; pagination `PaginatedResponse<T>`
  `{data[], pagination{page,pageSize,totalItems,totalPages,hasNextPage,hasPreviousPage}}`
  (transaction lists return the paginated object bare — the one exception). Field names:
  `transactionNumber`, `createdAt`, `recipientAzureTag`. The real spec is
  `../api/openapiv1.json` (19 paths, matches the code 1:1).
- **D4 — Versions: PINNED via CPM** (`backend/Directory.Packages.props`): net10.0, EF Core
  10.0.1, Identity/JwtBearer/OpenApi 10.0.1, FluentValidation 12.1.1, Mapperly 4.3.1,
  Scalar 2.12.4, YARP 2.3.0, Serilog.AspNetCore 10.0.0, xunit 2.9.3, Testcontainers 4.3.0,
  FluentAssertions 8.8.0. The EF9-vs-10 doc conflict is resolved in code.
- **D5 — Enum/casing: DECIDED.** Wire enums are **PascalCase strings** via a strict
  converter that rejects integers; DB stores strings via `HasConversion<string>()`.
  AzureTag: `^[a-z][a-z0-9_]{2,19}$`, unique. AccountNumber: **15 chars** `AB-XXXX-XXXX-XX`
  (both the "14" and "16" in older docs were wrong). Money DECIMAL(19,4).
  *Note: the built frontend still expects lowercase/snake and no envelope — reconciling the
  SPA to this contract is the tracked next milestone (frontend wiring).*
