# AzureBank

Personal bank account management system — a full-stack portfolio/study project:

- **`backend/`** — .NET 10 Web API (ASP.NET Core, EF Core 10, ASP.NET Identity + JWT,
  Argon2id step-up PIN, FluentValidation, Serilog, Mapperly, Scalar) with a dual-mode
  security architecture: direct stateless JWT **or** a YARP-based BFF
  (`AzureBank.Bff`) with HttpOnly cookie sessions where tokens never reach the browser.
- **`frontend/`** — React 19 + TypeScript + FluentUI v9 SPA (Redux Toolkit / RTK Query).
- **`docs/`** — architecture decision records, the design documentation the system was
  built from, and the real OpenAPI 3.1 specification generated by the running API.
- **`tests/`** — Bruno API collection and Schemathesis contract tests, alongside the
  xUnit suite in `backend/tests/`.

## Getting started

### Backend

```bash
cd backend
dotnet build AzureBank.slnx
dotnet test                     # 383 passing, 17 SQL-Server-only skips
```

The SQL-Server-gated proofs (idempotency single-execution under 24 parallel
requests, balance concurrency) run when `AZUREBANK_TEST_SQLSERVER` points to a
real database (LocalDB works; CI uses a mssql container):

```bash
AZUREBANK_TEST_SQLSERVER="Server=(localdb)\\MSSQLLocalDB;Database=AzureBankProofs;Trusted_Connection=True;TrustServerCertificate=True" \
  dotnet test --filter "Category=SqlServer"
```

Local run requires SQL Server (LocalDB works) and a JWT signing secret — see
[Secrets](#secrets). Create and seed the database with the Seeder tool:

```bash
dotnet run --project tools/AzureBank.Seeder -- reset --confirm
dotnet run --project tools/AzureBank.Seeder -- seed
dotnet run --project src/AzureBank.Api      # direct JWT mode
dotnet run --project src/AzureBank.Bff      # optional: BFF cookie mode
```

### Frontend

```bash
cd frontend
npm ci
npm run dev        # http://localhost:5173
npm run build
```

> Note: the SPA currently runs on internal mock data; wiring it to the real API
> (envelope + enum-casing reconciliation) is tracked as the next milestone.

## Secrets

No real secrets are committed. Local configuration goes in per-project
`appsettings.Development.json` files (git-ignored) — start from the committed
`appsettings.Development.json.example` templates — or in .NET user-secrets:

```bash
cd backend/src/AzureBank.Api
dotnet user-secrets init
dotnet user-secrets set "Jwt:Secret" "<random string, 64+ chars>"
dotnet user-secrets set "Idempotency:HashKey" "<random string, 32+ chars>"
dotnet user-secrets set "Security:PinPepper" "<random string, 32+ chars>"
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Server=(localdb)\\MSSQLLocalDB;Database=AzureBank;Trusted_Connection=True;TrustServerCertificate=True"
```

`Security:PinPepper` is a server-side secret mixed into the Argon2id PIN hash so a
stolen database can't brute-force the low-entropy PIN space
([ADR-0011](docs/adr/0011-pin-hash-pepper.md)); the API fails to start without it.
Use a cryptographically-random value (e.g. `openssl rand -base64 48`), not a typed
phrase. It supports zero-downtime rotation via a keyring
(`Security:PreviousPinPeppers`); when rotating, keep the **whole ring** (active +
previous) in a single secret provider — never in `appsettings.json`.

The Seeder reads the same connection string from its `appsettings.json` or the
`ConnectionStrings__DefaultConnection` environment variable, and must be given the
**same `Security:PinPepper`** (e.g. `Security__PinPepper` in its environment) —
otherwise seeded demo PINs cannot be verified.

## Idempotent monetary operations

Every mutating monetary endpoint (`POST /api/transactions/deposit`,
`/api/transactions/withdraw`, `/api/transfers`, `/api/transfers/internal`)
requires a client-generated **`Idempotency-Key`** header (UUID) and is
guaranteed to execute **at most once** per (user, endpoint, key) — under
client retries, concurrent duplicates, process crashes and database-retry
replays ([ADR-0009](docs/adr/0009-idempotency-monetary-operations.md)):

- Retry with the same key + same payload → the stored response is replayed
  byte-identically with `Idempotency-Replayed: true`.
- Same key + different payload → `422 IDEMPOTENCY_KEY_REUSE`.
- Duplicate still running → `409 IDEMPOTENCY_IN_FLIGHT` (retry shortly).
- Executed but the response was lost (crash) → `409 IDEMPOTENCY_RESULT_UNKNOWN`
  (verify via `GET /api/transactions`; never re-executed, never guessed).
- Errors (validation, insufficient funds, wrong PIN) release the key: fixing
  the payload and retrying the same key works.
- Keys are scoped per user and per endpoint; the idempotency window is 24h.

The guarantee is proven by tests: 24 byte-identical parallel requests with one
key → exactly one execution, balances mathematically exact, on EF InMemory
and on real SQL Server (3 deterministic rounds per run, also in CI).
